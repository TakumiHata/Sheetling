# プロジェクト指示書

## ロール
あなたは、複雑なドキュメント解析とデータ構造化のエキスパートエンジニアです。

## プロジェクト概要
Accessから出力された帳票PDFを、元の見た目（方眼レイアウト・色彩）を完全に維持したままExcel（.xlsx）に自動変換するシステムの開発。

## スコープと境界

### このリポジトリの責務
- **Input**: Access帳票PDF（.pdf）
- **Output**: LLMが認識・処理しやすい中間表現ファイル
    - **Markdown（.md）**: テキスト情報・論理構造（表、見出し、段落の関係性）
    - **JSON（.json）**: 座標・色彩・物理レイアウト情報（結合範囲、背景色、罫線、配置座標）

### Excel生成の流れ（このリポジトリの外側）
1. 本リポジトリが出力したMD + JSONを、**固定のプロンプトテンプレート**とともにLLMに渡す
2. LLMがopenpyxl等を用いたPythonコードを生成する
3. 生成されたPythonコードを実行してExcelファイル（.xlsx）を出力する

> **注意**: プロンプトテンプレートは帳票ごとに変わらず使い回す設計。帳票ごとに変わるのはMD/JSONデータのみ。

## プロジェクトの背景と最終目的
- **ビジネス課題**: ユーザーはAccessのレポート機能で生成された多様な帳票を、見た目（レイアウト、セル結合、色など）を維持したまま「方眼Excel」として出力したいと考えている。
- **技術的制約**: 
    - Accessから直接Excelへレイアウトを維持してエクスポートする標準機能は存在しない。
    - VBA等でのスクラッチ開発は保守性の観点から避けたい。
    - 出力される帳票フォーマットは複数存在し、事前定義が難しい（動的に変わる）。
    - LLMに直接PDFを渡してもExcel変換の精度は低い。高度なOCRツールなしでは高精度な変換は困難。
- **最終目的**: 既存の高度なPDF変換ソフト（ABBYY FineReaderなど）を代替するような、「PDFの視覚情報を動的に解析して、未知のレイアウトであっても自動で方眼Excelを生成する内製パイプライン」を構築すること。

## 要件（アーキテクチャ方針）
1. **方眼Excelレイアウト**: 
    - 出力先Excelを微小な正方形のセル（方眼紙）の集合として扱う。
    - PDF内の各要素の座標情報を基に、適切なセル範囲を結合・着色してレイアウトを再現する。
2. **動的解析**: 
    - テンプレート定義に頼らず、PDFの解析結果（Docling等を使用）から動的に構造を特定する。
3. **A4出力最適化**:
    - 生成されるExcelは、印刷時に**A4サイズ1ページ（または複数ページ）の幅に収まる**ように構成する。
    - 適切な余白調整（上下左右）を自動的に行い、印刷・PDF化が容易な設定をデフォルトとする。
4. **保守性**: 
    - Pythonをベースとした現代的なスタックを使用し、メンテナンスしやすい構成とする。

## 出力仕様

### Markdown（テキスト・論理構造）
LLMが「何がどこに書かれているか」を正確に理解するための論理情報を記述する。

- **見出し**: PDFの論理的なセクション区切りを `#` 見出しで表現
- **テーブル**: Markdownテーブル記法（`| col1 | col2 |`）で表構造を表現
- **テキスト**: 段落・箇条書き等の自然な文章構造を維持
- **要件**: NaN/Unnamed等のノイズは除去済みの状態で出力する

### JSON（座標・色彩・物理レイアウト）
LLMが「方眼Excelのどのセル範囲にどんなスタイルを適用するか」を判断するための物理情報を記述する。

想定されるスキーマの主要フィールド：
```json
{
  "page": {
    "width_pt": 595.28,
    "height_pt": 841.89,
    "grid_unit_pt": 10
  },
  "elements": [
    {
      "type": "rect | text | line",
      "bbox": { "x0": 0, "y0": 0, "x1": 100, "y1": 50 },
      "grid_bbox": { "col_start": 0, "row_start": 0, "col_end": 10, "row_end": 5 },
      "style": {
        "fill_color": "#FFFFFF",
        "border": { "top": true, "right": true, "bottom": true, "left": true },
        "border_color": "#000000"
      },
      "text": "サンプルテキスト",
      "font_size": 11
    }
  ]
}
```

### MD/JSONの紐付けルール
- JSONの各要素の `text` フィールドは、MDで抽出されたテキストと対応する
- **「MD = 論理の正」「JSON = 物理の参考」** が基本原則
- テキスト内容に矛盾がある場合、MD側を正とする

## LLM連携方針

### プロンプト設計
- プロンプトテンプレートは**帳票の種類によらず固定**
- テンプレートにMDとJSONを埋め込んで渡す構成
- LLMには「openpyxlを使ったPythonコード」を生成させる

### トークン制限への対策
- 10ptグリッドの場合、A4帳票1ページで約60列×84行 ≈ 5,040セル
- 大規模帳票ではMD+JSONだけで数千〜数万トークンに達する可能性がある
- 必要に応じて以下の対策を検討する：
    - 帳票を論理的なブロック単位に分割して個別に処理
    - JSONから冗長な情報（空セル等）を除外して圧縮
    - 繰り返しパターンの省略記法の導入

### 生成コードの品質管理
- LLM生成のPythonコードはサンドボックス環境（Docker等）で実行する
- 実行時エラーが発生した場合のリトライ戦略を設ける
- 出力Excelの基本的なバリデーション（ファイル破損チェック、シート存在確認等）を行う

## 技術的アドバイス・ベストプラクティス
1. **ハイブリッド解析の活用**:
    - **Docling**: レイアウト構造（表、セクション）の抽出に使用。
    - **pdfplumber**: 色彩情報（背景色、罫線）および幾何学座標の抽出に使用。
    - **MarkItDown**: テキスト情報の精密な抽出とMarkdown化に使用。
    - **ツール間の優先ルール**: テーブル構造は3ツールすべてが部分的にサポートするため、以下の優先順位で採用する：
        1. テキスト内容 → MarkItDown（最も精密）
        2. テーブル構造の認識 → Docling（構造解析に強い）
        3. 座標・色彩情報 → pdfplumber（幾何学情報に強い）
2. **座標の正規化とグリッド化**:
    - PDFのポイント座標を直接Excelのセルにマッピングするのではなく、最小公約数的なグリッド単位（例：2mm単位や一定のpt数）を算出し、正規化（スナップ）させることでレイアウトを安定させる。
3. **LLMの役割の限定**:
    - LLMにはバイナリ生成ではなく、抽出された構造データ（MD/JSON）に基づいたExcel生成のPythonコードを書かせることに集中させる。
